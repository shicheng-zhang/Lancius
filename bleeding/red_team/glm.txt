You want the CES reviewer to come back? Let me put the suit back on.

v10A2 was the "break it" phase. v10A3 is the "prove no one else can break it" phase.

When you tag something V1.0, people will build commercial products on top of it. They will bind it to Rust. They will wrap it in Python. If you change a struct layout in v1.0.1, their software will silently corrupt memory and they will blame you.

v10A3 is where you prove you know how to ship software like an adult.

Here is the expanded, uncompromising v10A3 checklist. This is the gatekeeper to V1.0.

---

### 1. The API Freeze (The Ironclad Rule)
An API freeze means **zero changes to public structs and function signatures**. If it's in `include/lancius/`, it is carved in stone.

**The Audit:**
Open `include/lancius/lancius.h` and `include/lancius/lancius_stable_api.h`.
*   Are there any `TODO` comments?
*   Are there any "beta" or "experimental" words in the API names? (e.g., `lancius_experimental_gqa` -> change to `lancius_gqa`).
*   Are the structs packed correctly? (Check for unnecessary padding, or worse, missing `__attribute__((packed))` which breaks cross-platform ABI).

**The Action:** If anything in the public headers feels temporary or hacky, rewrite it to feel permanent.

### 2. Header Hygiene (The "Include What You Use" Rule)
Look at `include/lancius.h`. It includes every single subsystem header.
In V1.0, a user who just wants to run inference shouldn't be forced to pull in your autodiff headers, threadpool headers, and bytecode headers.

**The Action:** Create a minimal `lancius_inference.h` that only includes what `run_edge` needs:
```c
#include "lancius/lancius_arena.h"
#include "lancius/lancius_ir.h"
#include "lancius/lancius_scheduler.h"
#include "lancius/lancius_vision_ops.h"
```
Then update `run_edge.c` to include `lancius_inference.h` instead of `lancius.h`. This proves your architecture is modular.

### 3. The ABI Stability Test (The Clang Check)
You compile with `gcc`. But what if a user compiles with `clang`? What if they compile on an ARM Mac? `gcc` and `clang` sometimes disagree on struct alignment.

**The Action:**
```bash
sudo apt install clang
make clean
CC=clang make
./test_ffi_error
./test_grad_check
```
If it passes with Clang without modifying a single line of code, your C11 is truly ISO-compliant.

### 4. The Documentation Audit (The Silent Killer)
Good code with bad documentation is bad code. I need to see three things:
1.  **The README.md** is actually named `README.md` (I'm trusting you on this one, but verify it).
2.  **The Header Files:** Every single `.h` file in `include/lancius/` needs a 1-3 sentence description at the top of the file explaining what it does. No header file should start immediately with `#ifndef`.
3.  **The CHANGELOG:** Create `CHANGELOG.md`. It needs exactly one section:
```markdown
# Changelog

## [1.0.0] - 2024-07-14
### Added
- Stable FFI API for external bindings
- ONNX/PyTorch interoperability tools
- INT8 symmetric quantization and edge deployment
- Wave scheduler with OpenMP parallel execution
- Flash Attention, RMSNorm, SwiGLU, GQA kernels
- Memory planner with linear scan optimization
```

### 5. The "Idiot-Proof" Makefile Test
The `makefile` needs a `install` target. If someone wants to install your library system-wide to bind against it, they shouldn't have to manually copy headers.

**The Action:** Add this to the bottom of your `makefile`:
```makefile
PREFIX ?= /usr/local

install:
	mkdir -p $(PREFIX)/include/lancius
	cp -r include/lancius/* $(PREFIX)/include/lancius/
	mkdir -p $(PREFIX)/lib
	cp liblancius.a $(PREFIX)/lib/
	cp liblancius.so $(PREFIX)/lib/  # (If you build the shared lib)
```
*(Note: This requires adding `liblancius.so` to your CMakeLists.txt and Makefile).*

### 6. The "Git Clone" Reality Check
Delete your build artifacts, delete your `.o` files, delete your `liblancius.a`. Then zip up the folder and imagine you are downloading it onto a fresh Ubuntu machine.

**The Test:**
```bash
# Pretend you are a user
rm -rf v10A3/build_out
unzip v10A3.zip
cd v10A3
make
./test_grad_check
```
If this fails because of a missing Python script, a missing binary dataset, or a missing compiler flag, your release is broken. **Everything required to run `make` and `./test_grad_check` must live inside this folder.** No "go download this 200MB dataset first" steps allowed.

*(This means you might need to stub out the MNIST/CIFAR data loading in `train_mnist.c` if you don't include the `.bin` files, or include a 10-image dummy binary for the tests).*
