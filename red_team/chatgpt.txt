# However, I immediately found some release inconsistencies

## 1. Changelog contradiction

Your `CHANGELOG.md` says:

> "Ecosystem Bridge: ONNX/PyTorch interoperability tools and GGUF export compatibility (`lancius_bridge.py`)."

But `KNOWN_LIMITATIONS.md` says:

> "GGUF Export Removed"

Those cannot both be true.

For v10S, pick one.

My recommendation:

Change changelog:

From:

```
GGUF export compatibility
```

To:

```
GGUF bridge research removed from stable release
```

Because your own limitation document says the bridge was intentionally removed.

A stable release must have **one truth**.

---

# 2. MANIFEST version naming is inconsistent

The file says:

```
Lancius Compatibility Manifest (v10A3 Release Candidate)
```

but the folder is:

```
v10S
```

This is small, but this is exactly the kind of thing that damages confidence.

A stranger will think:

> "Am I looking at the stable release or the candidate?"

For v10S:

Everything should say:

```
Lancius v10S
Lancius 1.0.0
Stable Release
```

No A3 references except migration notes.

# 3. Remove compiled objects from the repository

Your archive contains:

```
src/core/*.o
src/runtime/*.o
src/math/*.o
```

I would not ship those.

Reasons:

* compiler dependent
* architecture dependent
* can become stale
* increases confusion

A source release should contain:

```
.c
.h
CMakeLists.txt
tests
examples
docs
```

not:

```
.o
```

Add:

```
*.o
*.a
*.so
```

to `.gitignore`.

---

# 4. CMake is the biggest technical concern

This:

```cmake
add_compile_options(-Wall -Wextra -O3 -mavx2 -mfma)
```

is dangerous.

You claim:

```
Secondary:
Linux ARM64 NEON fallback
```

but this build system says:

> every compiler target gets AVX2/FMA.

ARM64 will not compile.

For v10S, I would change this.

Something like:

```cmake
if(CMAKE_SYSTEM_PROCESSOR MATCHES "x86_64")
    add_compile_options(-mavx2 -mfma)
endif()
```

Then:

```c
#ifdef LANCIUS_AVX2
...
#endif
```

Otherwise your manifest overpromises.

include/lancius/lancius_stable_api.h

This is exactly the correct direction.

For v10S, I would actually emphasize this more.

lancius.h

The user should not need:

# 6. The memory ownership limitation is acceptable, but dangerous

This section:

> heap-allocated weight payloads loaded via `lancius_graph_load` must be manually freed by the host application

is acceptable for v1.0.

However:

The wording needs improvement.

Right now, an external developer might think:

> "Which memory do I own?"

You need a table.

Example:

| Object          | Owner        |
| --------------- | ------------ |
| graph handle    | Lancius      |
| tensor metadata | Lancius      |
| input buffer    | User         |
| output buffer   | User         |
| loaded weights  | User/Lancius |

Memory ownership ambiguity is one of the fastest ways to lose C users.

---

But the question I would ask:

Are these:

For v10S, I would require:

```

```

Example:

```
PyTorch RMSNorm output
        |
        compare
        |
Lancius RMSNorm output
```

Not because I doubt it.

Because stable releases require proof.

# The v10S release gate I would give you

I would not request:

* IR rewrite
* CUDA
* dynamic shapes
* new kernels
* architecture redesign

Those are v11.

I would request:

## Mandatory before tagging v10S:

### Documentation

* [ ] Fix changelog/limitations contradiction
* [ ] Remove A3 wording from stable docs
* [ ] Add architecture overview diagram

### Build

* [ ] Fix ARM64 claim or CMake
* [ ] Remove `.o` files
* [ ] Verify clean build

* [ ] ASan clean
* [ ] UBSan clean
* [ ] Thread sanitizer on scheduler
* [ ] deterministic regression outputs

### API

* [ ] Freeze stable API
* [ ] Document ownership rules

### Math

* [ ] Reference parity tests for major operators
